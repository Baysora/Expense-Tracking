# Baysora Expense Tracking

Multi-tenant expense tracking platform for HoldCo/OpCo organisations. Built on Azure.

## Architecture

| Component | Technology |
|---|---|
| Frontend | React + TypeScript + Vite (hosted on Azure Static Web Apps) |
| API | Azure Functions v4 (TypeScript/Node.js 20) |
| Database | Azure SQL Database via Prisma ORM |
| File Storage | Azure Blob Storage (private, SAS URLs) |
| Auth | Azure AD B2C (email + password) |
| IaC | Azure Bicep |
| CI/CD | GitHub Actions |

## User Roles

| Role | Access |
|---|---|
| `HOLDCO_ADMIN` | Full control: manage all OpCos, users, categories, attachment rules; view and approve all expenses; submit own expenses |
| `HOLDCO_USER` | Submit expenses under the HoldCo internal entity; no approval rights |
| `OPCO_ADMIN` | Manage users within their OpCo; view all OpCo expenses; approve/reject expenses |
| `OPCO_MANAGER` | Approve or reject submitted expenses within their OpCo |
| `OPCO_USER` | Submit expenses with receipts and invoices |

## Supported File Types

Receipts and invoices: **PDF, JPEG, PNG, WEBP, HEIC, HEIF, TIFF, GIF, BMP** (max 10 MB each)

## Local Development

### Prerequisites

- **Node.js 20+**
- **Docker** — for a local SQL Server instance
- **Azurite** *(optional)* — only needed for attachment upload/download; the rest of the app works without it

> Azure Functions Core Tools are **not required** for local development. The API runs via a lightweight Express server that emulates the Azure Functions runtime.

### 1 — Install dependencies

```bash
npm ci
npm run build --workspace=packages/shared
```

### 2 — Start a local SQL Server

```bash
docker run -d \
  --name sql-local \
  -e ACCEPT_EULA=Y \
  -e SA_PASSWORD=YourPass123! \
  -p 1433:1433 \
  mcr.microsoft.com/mssql/server:2022-latest
```

Wait ~10 seconds for SQL Server to start before continuing.

### 3 — Configure the API

```bash
cp api/local.settings.json.example api/local.settings.json
```

The example file is already configured for the Docker SQL Server above. Open `api/local.settings.json` and verify or adjust:

```json
{
  "IsEncrypted": false,
  "Values": {
    "FUNCTIONS_WORKER_RUNTIME": "node",
    "DATABASE_URL": "sqlserver://localhost:1433;database=ExpenseTracking;user=sa;password=YourPass123!;trustServerCertificate=true",
    "JWT_SECRET": "local-dev-secret-change-in-production",
    "DEV_MODE": "true",
    "AZURE_STORAGE_CONNECTION_STRING": "",
    "AZURE_STORAGE_CONTAINER_NAME": "expense-attachments"
  }
}
```

> **`DEV_MODE=true`** bypasses Azure AD B2C. Authentication is handled with a simple `Bearer dev:<base64(email:password)>` header generated by the login page.

### 4 — Apply the database schema and seed data

```bash
cd api
npx prisma db push
npx prisma db seed
cd ..
```

### 5 — (Optional) Start Azurite for local Blob Storage

Required only if you want to test file attachment upload and download.

```bash
npx azurite --location .azurite --debug .azurite/debug.log &
```

Then update `AZURE_STORAGE_CONNECTION_STRING` in `api/local.settings.json` to:

```
UseDevelopmentStorage=true
```

### 6 — Start the API and frontend

In two separate terminals:

```bash
# Terminal 1 — API (runs on http://localhost:7071)
npm run dev:api

# Terminal 2 — Frontend (runs on http://localhost:5173)
npm run dev:frontend
```

Then open **http://localhost:5173**.

## Azure Deployment

### 1. Provision Azure Resources

```bash
az group create --name rg-expense-tracking-dev --location eastus

az deployment group create \
  --resource-group rg-expense-tracking-dev \
  --template-file infra/main.bicep \
  --parameters infra/parameters/dev.bicepparam \
  --parameters sqlAdminPassword="YourSecurePassword123!"
```

### 2. Configure Azure AD B2C

1. Create an Azure AD B2C tenant
2. Register two app registrations: `expense-tracking-api` and `expense-tracking-spa`
3. Create a Sign up / Sign in user flow
4. Add custom attributes: `extension_role` (String) and `extension_opCoId` (String)
5. Include these attributes in token claims

### 3. Set GitHub Secrets

In your repository settings → Secrets and variables → Actions:

```
AZURE_STATIC_WEB_APPS_API_TOKEN   # From Azure Portal → Static Web App → Manage token
DATABASE_URL                       # From Bicep outputs
AZURE_STORAGE_CONNECTION_STRING    # From Bicep outputs
B2C_TENANT_NAME                    # Your B2C tenant name
B2C_POLICY_NAME                    # B2C_1_signupsignin
B2C_CLIENT_ID                      # SPA app registration client ID
B2C_AUDIENCE                       # API app registration client ID
B2C_API_SCOPE                      # https://{tenant}.onmicrosoft.com/expense-api/access_as_user
```

### 4. Run Database Migrations

```bash
DATABASE_URL="sqlserver://..." npx prisma db push
DATABASE_URL="sqlserver://..." npx prisma db seed
```

### 5. Push to main

```bash
git push origin main
```

GitHub Actions will build and deploy automatically.

## Branding

Brand colors are defined in `frontend/src/styles/tokens.css` as CSS custom properties. Update `--color-primary`, `--color-accent`, etc. once Baysora brand colors are confirmed. All UI components reference these tokens, so a single file update applies the brand globally.

## Project Structure

```
/
├── packages/shared/       # Shared TypeScript types (enums, interfaces)
├── api/                   # Azure Functions (TypeScript)
│   ├── prisma/           # Database schema and seed data
│   └── src/
│       ├── functions/    # HTTP trigger functions (grouped by domain)
│       └── lib/          # Auth, Prisma client, Blob Storage, error helpers, local dev server
├── frontend/              # React SPA (Vite + TypeScript)
│   └── src/
│       ├── pages/        # Route components per role (holdco/, opco/, user/)
│       ├── components/   # Shared UI components
│       └── lib/          # Auth context, API client, router, utilities
├── infra/                 # Azure Bicep IaC templates
│   └── modules/          # SQL, Storage, SWA, Function App modules
└── .github/workflows/    # GitHub Actions CI/CD
```
